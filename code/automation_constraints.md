# 电商平台商家后台自动化 - 经验约束日志（综合版）

> 创建时间: 2026-05-09
> 来源平台: 京东商家后台（京麦）、抖音商家后台（抖店）
> 任务类型: 每日自动登录多个子账号，抓取账户余额和店铺保证金并保存到Excel

***

## 零、约束分级与时效管理

### 约束分级定义

| 级别 | 含义 | AI 行为 |
|------|------|---------|
| **P0 - 硬约束** | 平台强制要求，已验证不可绕过 | 必须遵守，不可偏离 |
| **P1 - 强约束** | 经验证最优路径，绕过成本高 | 优先使用，失败后可尝试探索 |
| **P2 - 弱约束** | 经验提示，可能有时效性 | 仅供参考，AI 可自行判断是否尝试新方案 |

### 各维度约束分布

| 维度 | P0 | P1 | P2 |
|------|:--:|:--:|:--:|
| 浏览器环境 | headless禁用、Edge必须 | 分辨率固定 | UA版本 |
| 登录态管理 | 登录态失效多形态检测 | cookie复用不可行 | 信号文件协调 |
| 页面导航 | URL+内容双重验证、触发方式确认 | SPA等待策略 | 左侧菜单兜底 |
| 弹窗处理 | 区分促销弹窗与业务弹窗 | hover面板保护、z-index扫描、Shadow DOM探测 | ESC兜底 |
| 元素定位 | 禁止硬编码坐标 | locator API优先、鼠标轨迹模拟 | 区域过滤兜底 |
| 数据提取 | 数据来源固定、跨页面对账防投毒 | 格式差异处理 | 眼睛图标点击 |
| 反检测 | webdriver/plugins/chrome、Canvas/WebGL指纹、屏幕/显卡参数对齐 | 启动参数、WebRTC泄漏 | CDP连接、请求频率 |
| 多店铺切换 | 必须重新打开弹窗、Canvas指纹进程隔离 | 导航回目标页、重定向循环防护 | 滚动加载验证 |
| 异常处理 | 单账号失败不中断流程 | evaluate崩溃保护、连续失败熔断 | 截图保存 |

### 约束失效检测

当以下情况发生时，AI 应主动进入探索模式重新验证约束的有效性：
- 同一个 P1/P2 约束导致连续失败 >= 3 次
- 平台 UI 明显更新（用户提示或截图变化）
- 距离上次验证超过 30 天（P1）或 90 天（P2）

### 探索模式触发机制

**被动触发（自动）：** 见上方"约束失效检测"，满足任一条件即自动进入探索模式。

**主动触发（用户信号）：**
当用户提示中包含以下关键词时，AI 应进入探索模式：
- "重新验证" / "重新检查"
- "尝试新方案" / "换一种方式"
- "约束可能变了" / "规则可能过时了"
- "优化"（明确指向约束或流程而非代码时）

进入探索模式后，AI 应在执行前输出：
> 探索模式已激活，将对 P1/P2 约束进行重新验证。P0 硬约束保持不变。

## 一、浏览器环境约束

### 必须验证

- [ ] 浏览器必须使用系统已安装的Edge，不可使用Playwright自带的Chromium
- [ ] 必须为可见模式（headless=False），因为需要人工处理验证码
- [ ] 分辨率必须固定为常见桌面分辨率，不可随机生成
- [ ] user\_agent需与实际使用的浏览器版本匹配，不可写死版本号

### 禁止假设

- [x] 禁止假设无头模式能正常工作 --> 目标平台会检测并阻断
- [x] 禁止假设viewport=None能自适应 --> 必须固定分辨率，否则指纹异常
- [x] 禁止假设系统Edge版本固定 --> 需运行时获取实际版本号

### 已知坑

- 坑1: 使用系统Edge时不需要安装Playwright自带浏览器，但需要确认Edge已安装
- 坑2: Playwright的channel参数会直接调用系统Edge，但默认会注入自动化标记

## 二、登录约束

### 必须验证

- [ ] 登录页面存在账号密码/手机号输入框，需确认占位符文本以正确定位输入元素
- [ ] 登录成功后URL域名会变化，需在首次登录后确认当前环境的登录后域名特征
- [ ] 验证码等待必须有超时机制，不可无限等待
- [ ] 登录态文件存在不等于登录态有效，需要页面内容验证（见下方触发条件）
- [ ] 登录态可能"看起来"有效但实际权限已过期（页面显示权限调整提示），需单独检测该类文本

### 登录态失效的多种形态

| 失效类型 | 页面表现                      | 检测方式   | 恢复方式                          |
| ---- | ------------------------- | ------ | ----------------------------- |
| 会话过期 | 跳转到登录页，URL含登录路径           | URL检查  | 正常重新登录                        |
| 权限调整 | 页面显示"权限被调整无法访问"，URL无登录关键字 | 页面文本检测 | 必须清除旧cookie再登录，否则登录后立即跳回权限错误页 |
| 首次登录 | 正常登录页面                    | 同上     | 正常登录流程                        |

### 触发条件

此约束在以下场景下生效:

- 首次登录新账号
- 长时间未登录的账号（登录态有效期通常较短，跨天或几小时后过期）
- 短时间内频繁切换多个账号（可能触发安全策略升级）
- 权限调整导致的强制退出不可预测

### 不可行方案

- 尝试过自动识别滑块验证码 --> 目标平台验证码复杂度高，第三方打码平台成功率低且成本高
- 尝试过保持登录状态(cookie复用) --> 资金等敏感数据在独立子域名下，需要独立认证，主站cookie不通用
- 尝试过降低验证码频率 --> 平台安全策略严格，无法通过常规手段绕过
- 仅检查URL是否包含登录关键字来判断是否需重新登录 --> 权限错误时URL无登录关键字，会漏判
- 不删除旧状态文件直接重新登录 --> 旧cookie会干扰，导致登录后立即跳回权限错误页

### 已知坑

- 坑1: 验证码可能在输入账号后、输入密码后、点击登录后任意时刻弹出，不是固定节点
- 坑2: 不同账号可能触发不同类型的验证（滑块/短信/图形），脚本需能识别并处理多种情况
- 坑3: 使用旧cookie打开页面后会先短暂显示正常页面再跳转到登录页，中间存在竞态条件
- 坑4: 新浏览器上下文创建时必须重新注入反检测脚本，旧上下文的注入脚本不会自动带到新上下文

## 三、页面导航约束（核心教训）

### 3.1 SPA路由与直接URL访问

**必须验证:**

- [ ] 目标数据页面是否在独立子域名或独立模块下 --> 如果是，直接URL访问可能进入错误模块
- [ ] 同一域名下不同路径可能对应完全不同的业务模块，需确认路径的精确性
- [ ] 导航到目标功能页面后必须验证URL是否确实到达目标地址，不能仅以goto成功为准
- [ ] 导航失败跳转到登录页说明登录态已过期，不是路径错误

**禁止假设:**

- [x] 禁止假设可以通过URL直接访问数据页面 --> 可能进入同名但不同业务模块的页面
- [x] 禁止假设URL路径中包含某个关键词就对应目标功能 --> 路径层级不同可能导致进入完全不同的子系统
- [x] 禁止假设导航成功后页面内容一定正确 --> 必须同时验证URL和页面内容

**已知坑:**

- 坑1: 该平台存在多个"财务"相关模块，父级路径相同但子路径不同，直接访问父路径会默认进入第一个子模块（可能是客服财务而非店铺财务），导致获取到完全错误的数据
- 坑2: 错误模块中的页面结构和字段名称可能与正确模块高度相似（都有"账户余额"和"店铺保证金"字段），但数据含义完全不同，仅凭字段名无法区分是否进入了正确页面
- 坑3: 从错误模块获取的数据可能看起来格式正确（金额、数字），但数值完全错误，且这种错误无法通过数据校验发现
- 坑4: SPA页面跳转后URL可能不立即更新，需要等待并验证
- 坑5: 页面在导航过程中执行evaluate会抛出执行上下文被销毁的错误，需要在关键操作处加异常保护

**不可行方案:**

- 直接URL访问父级财务路径 --> 会进入默认子模块，而非目标子模块
- 不验证最终URL就认为导航成功 --> 可能被重定向到登录页或其他模块

**必须验证的条件:**

- [ ] 进入数据页面后，必须验证页面标题或关键文本确认进入了正确的模块
- [ ] 不可仅凭URL判断进入了正确页面，必须结合页面内容验证

### 3.2 左侧菜单导航

**必须验证:**

- [ ] 左侧菜单是SPA单页应用的导航方式，点击菜单项不会触发页面刷新，而是局部更新内容区
- [ ] 不同店铺的页面布局宽度可能不同，导致同一菜单项在不同店铺中的屏幕坐标不同
- [ ] 左侧菜单的类名可能动态变化，菜单展开需要点击触发器且展开状态不稳定
- [ ] 确认触发方式：hover 还是 click，不可假设；搞反了目标元素根本不会出现

**已知坑:**

- 坑1: 部分店铺的页面宽度比其他店铺窄，左侧菜单项的水平坐标可能差异巨大（如正常店铺x坐标为12，窄页面店铺x坐标为190），基于固定坐标范围的点击策略会失效
- 坑2: 在窄页面中，JS的精确文本匹配+坐标限制可能找到元素并成功触发点击事件，但页面未发生跳转（可能是点击到了被遮挡的元素或非交互层）
- 坑3: 页面加载等待策略对成功率影响极大，等待策略不当会导致超时失败
- 坑4: 左侧菜单点击展开可能不生效（CSS类名动态变化），直接URL导航反而更可靠
- 坑5: 用 click 去触发 hover 菜单 --> 下拉面板根本不会出现，因为触发机制完全不同

**禁止假设:**

- [x] 禁止假设所有店铺的页面布局完全一致 --> 同一平台不同店铺可能有不同的页面宽度
- [x] 禁止假设JS点击事件触发就等于导航成功 --> 需要验证页面内容确实发生了变化
- [x] 禁止假设点击后页面会立即跳转 --> SPA应用的页面切换可能需要等待异步数据加载
- [x] 禁止假设左侧菜单在所有场景下都可用 --> 类名/状态不稳定的情况下URL导航是更好的兜底
- [x] 禁止假设所有交互都是 click 触发 --> 部分下拉菜单/面板是 hover 触发，搞反了元素根本不会出现

### 3.3 页面加载等待策略

**必须验证:**

- [ ] 等待策略需根据页面特征选择，不同等待策略的适用场景不同

**已知坑:**

- 坑1: 使用"等待网络空闲"策略在某些店铺会超时失败，因为页面可能持续有后台请求（如WebSocket、轮询），导致网络永远不会完全空闲
- 坑2: 使用"等待DOM内容加载完成"策略更稳定，但可能页面数据尚未渲染完成，需要额外等待
- 坑3: 两种等待策略各自有失效场景 -- 网络空闲可能因持续后台请求永不触发；DOM加载完成可能数据尚未渲染。单独依赖任一策略都不可靠
- 坑4: 页面内容出现不等于数据就绪，异步渲染的数据可能在DOM加载完成后才填充。必须验证目标数据是否实际出现，而非仅验证页面结构
- 坑5: 页面可能先加载框架再异步加载数据，等待时间需充足（切换店铺后约8-15秒）

**不可行方案:**

- 对所有页面统一使用"等待网络空闲" --> 部分页面会因持续的后台请求而永远无法达到网络空闲状态

## 四、弹窗处理约束（抖店重点）

### 4.1 弹窗类型识别

**必须验证:**
- [ ] 弹窗关闭逻辑必须区分"促销弹窗"和"功能性弹窗"，否则会把店铺选择器等业务弹窗一同关闭
- [ ] 关闭弹窗前需验证弹窗类型：业务弹窗（如店铺选择器）通过特定class前缀可识别，不应被通用关闭逻辑处理
- [ ] 弹窗关闭后需验证目标元素是否仍可交互，不能仅以"关闭动作执行了"为判断标准
- [ ] 弹窗深度检测：仅靠 class 识别不够，需引入 z-index 扫描识别最顶层遮罩元素
- [ ] 若常规 CSS 选择器无法触及弹窗内容，需探测 Shadow Root 结构（2026年促销弹窗已大量采用 Shadow DOM 封装）

### 4.2 弹窗干扰

**已知坑:**

- 坑1: 在循环中每次打开店铺选择器前调用通用弹窗关闭函数，会破坏hover触发下拉菜单的页面状态，导致下拉菜单中关键按钮消失
- 坑2: 弹窗关闭时若页面正在导航，evaluate会因执行上下文被销毁而崩溃，需要全局try/except保护
- 坑3: 618等运营促销弹窗在首页频繁出现，全屏遮罩层反复弹出，首页操作极度不稳定
- 坑4: 弹窗遮罩层可能有多个嵌套，需要排除业务弹窗内的遮罩元素
- 坑5: 关闭按钮的文本可能存在于父元素而非直接子节点，需匹配直接文本节点
- 坑6: 2026年促销弹窗大量采用 Shadow DOM 封装，常规 CSS 选择器无法触达内部元素，需通过 element.shadowRoot 穿透

**不可行方案:**

- 在循环中每次切换店铺前调用通用弹窗关闭函数 --> 会破坏下拉面板
- 试图在首页完成店铺切换 --> 运营弹窗干扰严重
- 用display:none强制隐藏业务弹窗 --> 会导致后续hover面板状态异常

### 触发条件

- 登录后首次进入后台页面时会弹出运营促销弹窗
- 切换店铺后页面加载过程中可能触发新的促销弹窗

## 五、元素定位与点击约束

### 5.1 坐标依赖问题

**必须验证:**

- [ ] 目标元素在不同店铺/不同会话中的坐标是否稳定
- [ ] 是否存在页面宽度/布局差异导致坐标漂移的情况

**禁止假设:**

- [x] 禁止假设目标元素的坐标在所有情况下都相同 --> 页面宽度、菜单展开状态等都会影响坐标
- [x] 禁止假设JS点击事件触发就等于用户操作成功 --> 需验证页面实际响应
- [x] 禁止假设hover和click的触发方式相同 --> 抖店的店铺切换下拉菜单是hover触发，不是click触发

**已知坑:**

- 坑1: 基于坐标的点击策略在页面布局不一致时会失效，需要准备不依赖坐标的备用方案
- 坑2: 使用JS精确文本匹配时，如果坐标范围设置过窄，可能在某些页面中找不到目标元素
- 坑3: hover触发后下拉菜单出现需要等待（约2-3秒），立即查找会找不到目标元素

### 5.2 Playwright定位API优先原则

**必须验证:**

- [ ] 使用Playwright locator API优先于裸evaluate JavaScript查找元素
- [ ] 元素可能被弹窗遮挡，需使用force参数绕过指针事件拦截
- [ ] 元素查找需要多种策略兜底：class选择器 -> 文本匹配 -> 区域位置过滤

**已知坑:**

- 坑1: JavaScript正则语法不能在Python字符串中直接使用，需用字符串形式传递
- 坑2: SVG元素的className是SVGAnimatedString而非普通字符串，调用字符串方法会报错
- 坑3: 元素可能有多层嵌套，textContent包含子元素文本导致匹配不精确
- 坑4: 页面加载时序问题导致元素在DOM中存在但不可交互

### 5.3 定位策略的失效边界

**单一策略无法覆盖所有场景，因为:**

- 坐标定位：页面宽度变化、菜单展开状态变化会导致坐标漂移
- 文本匹配：可能匹配到非目标元素，不同店铺可能有同名文本
- 容器内定位：需要已知容器特征，新页面/新版本可能改变DOM结构
- 叶节点文本查找：可能匹配到非交互元素（纯展示文本）
- URL直接导航：可能进入同名但不同的业务模块，必须验证页面内容

**关键约束:**

- [ ] 任何策略的成功都需要验证 -- 点击触发不等于导航成功，必须检查页面内容变化
- [ ] 低精度策略匹配到多个候选元素时，需按位置/结构筛选，不可随机选择

**已知坑:**

- 坑1: 高精度策略找到元素并触发点击，但页面未响应，此时不能认为成功，必须验证页面内容变化
- 坑2: 低精度策略可能匹配到多个候选元素，需要按位置或其他特征排序，优先尝试最可能的那个
- 坑3: URL直接导航作为最终兜底时，路径必须精确到目标子模块，使用父路径可能进入错误模块

### 5.4 误点击防护

**必须验证:**

- [ ] 页面中是否存在与目标文本相似的其他元素（如"找客服"与"财务"位置接近）
- [ ] 点击目标是否在预期的页面区域（如左侧菜单栏），需限制坐标范围避免点到其他区域

**已知坑:**

- 坑1: 页面中可能存在多个包含相似文本的元素，简单的文本匹配可能点到错误元素
- 坑2: 需要验证目标元素的父容器是否在预期的导航区域（如侧边栏），避免点到内容区域中的同名文本

## 六、数据获取约束

### 6.1 余额显示交互

**必须验证:**

- [ ] 余额数据默认可能是隐藏状态（显示为星号或占位符如`¥-`），需要触发特定交互才能显示
- [ ] 触发显示的交互元素（如眼睛图标）位置在余额文本附近，需通过相对位置关系定位

**已知坑:**

- 坑1: 余额隐藏时页面文本中不包含实际数字，正则提取会失败或提取到占位符
- 坑2: 眼睛图标是SVG元素，定位方式与普通HTML元素不同，需要遍历所有SVG元素并通过坐标距离判断

### 6.2 数据格式差异（抖店/字节系平台）

**必须验证:**

- [ ] 同一平台不同页面的数据格式可能完全不同：账户中心使用半角¥符号+千分位逗号分隔，首页使用全角￥符号+万单位
- [ ] 提取保证金数据时需区分不同类型的保证金（基础保证金 vs 体验保证金），两者格式相同但含义不同
- [ ] 页面文本中存在相似的字段名（如"聚合账户余额"会干扰"账户余额"匹配），需要精确排除

**已知坑:**

- 坑1: 从首页提取账户数据格式不一致且数据可能不完全
- 坑2: 仅匹配"账户余额"而不排除"聚合"变体，会匹配到错误行
- 坑3: "体验保证金"可能有多行变体（如"体验保证金(可提现)"），需要排除含括号的变体

**边界情况:**

- 金额为0时显示为`¥0.00`
- 保证金可能为0或为空
- 多个地方的"账户余额"文本（如聚合账户、子账户）

### 6.3 数据提取验证

**必须验证:**

- [ ] 提取到的金额数据必须做合理性检查（如金额不能为负数、不能超过合理范围）
- [ ] 正则提取可能匹配到页面中其他数字，需确保正则模式足够精确

**禁止假设:**

- [x] 禁止假设正则提取到的第一个数字就是目标数据 --> 页面中可能存在多个数字
- [x] 禁止假设余额显示后立即可以提取 --> 可能需要等待动画或异步渲染完成
- [x] 禁止假设从首页提取的金额格式和账户中心一致 --> 不同页面数据格式完全不同

### 6.4 跨页面对账防"数据投毒" (P0)

**必须验证:**
- [ ] 在首页/概览页提取的"模糊总额"应与子页面提取的"精确余额"进行逻辑校验（如首页显示 1.5 万，子页面显示 15,230.50 元，两者应在合理偏差内）
- [ ] 如果两个页面的金额偏差超过合理阈值（如超过 5%），应视为数据异常，立即在备注中标记

**触发原因:** 平台有时会针对爬虫返回虚假/随机的金额数字（反爬策略之一），通过多页面对账可识别数据是否被"投毒"。仅从单一页面提取的数据无法自我校验真伪。

**已知坑:**
- 坑1: 两页面数据格式不同时（如首页用"万"单位、子页面用精确数值），需要先统一单位再比较
- 坑2: 该约束只在两个页面都有数据时生效；若只能获取单一来源，需在备注中注明"未对账"

## 七、反检测约束

### 7.1 已知检测维度

**必须覆盖的攻击面:**

| 检测维度                | 被检测后果          | 覆盖要求                 |
| ------------------- | -------------- | -------------------- |
| navigator.webdriver | 最基础的自动化检测      | 不能为true，必须为undefined |
| 自动化启动标记             | 浏览器默认添加        | 必须通过参数移除             |
| navigator.plugins   | 无头浏览器plugins为空 | 长度不能为0，需注入常见浏览器插件    |
| window.chrome对象    | 自动化工具中缺失或结构不完整 | 对象必须存在且结构完整          |
| 内核自动化特征             | 浏览器内核的自动化标记    | 必须通过启动参数禁用           |
| 请求频率                | 短时间大量请求        | 操作间隔不能太短，需模拟人类节奏     |
| 设备指纹一致性             | 多个参数互相矛盾       | 所有指纹参数必须逻辑自洽         |

### 7.2 高级指纹对抗（2026年主流平台已全面引入）

**Canvas/WebGL 指纹 (P0):**
- 必须在浏览器初始化阶段注入一致的 Canvas/WebGL 指纹，不可使用 Playwright 默认空指纹
- 多个账号在同一 browser_context 下运行会共享 Canvas 指纹，详见第八章环境隔离约束

**屏幕/显卡参数对齐 (P0):**
- window.screen 的 availHeight、colorDepth 以及显卡的 unmaskedRenderer 必须与模拟的浏览器 UA 保持逻辑一致
- 已知坑：如果 UA 声明为 Windows 11 + Edge，但 WebGL 返回的渲染器是 Linux 环境常见的软件渲染（如 SwiftShader），这种"指纹逻辑矛盾"会瞬间触发 P0 级风控告警
- 必须验证：WebGL RENDERER 字符串是否与操作系统/浏览器品牌匹配，不可出现跨 OS 的渲染器名称

**鼠标轨迹生物识别 (P0):**
- 禁止使用纯直线模拟点击：Playwright 的 click() 虽有默认延迟，但其轨迹在服务端视角仍是"点对点"的瞬间位移
- 必须引入贝塞尔曲线或高斯分布的随机轨迹模拟，操作间隔不可均匀固定
- 必须验证: 连续操作的时间间隔是否有合理波动（非固定值），鼠标移动路径是否有曲率

**WebRTC 泄漏检测:**
- 必须检查 navigator.mediaDevices：自动化环境常返回空设备列表，是极强的"非真人"信号
- 需注入合理的设备枚举结果或通过浏览器参数禁用 WebRTC

**CDP (Chrome DevTools Protocol) 连接检测 (P2，抖店重点):**
- 抖音系平台对 CDP 连接非常敏感，长时间保持 Debugger 挂载可能触发风控
- 避免长时间保持 Debugger 挂载，在非必要操作期间尽量减少不必要的 evaluate 调用批量操作

### 7.3 不可行方案

- 只设webdriver为undefined --> plugins、chrome对象等维度仍暴露
- 只禁用内核自动化特征 --> 需配合移除默认启动参数才有效
- 随机化窗口大小 --> 反而增加不稳定性，固定常见分辨率更自然
- 用 Playwright 默认 click() 不做轨迹干预 --> 服务端可识别为机械操作
- 多个账号共享同一个 browser 实例的 Canvas 指纹 --> 关联封号风险

**约束: 反检测配置需在每次新平台接入时重新验证**，不可直接复用其他平台的参数清单。

## 八、多店铺/多账号处理约束

### 8.1 账号隔离性

- [ ] 每个账号必须使用独立的页面或context，完成后清除所有cookies
- [ ] 上一个账号的登录态不能泄漏到下一个账号
- [ ] 禁止在同一个context中连续登录多个账号而不清除状态
- [ ] Canvas 指纹污染隔离 (P0)：多个账号在同一 browser_context 下运行会共享 Canvas 指纹，即使清除了 Cookie 也无法消除此关联。建议每切换 3-5 个账号，彻底关闭并重新拉起一个新的浏览器进程，而非仅换 context

### 8.2 店铺切换流程（抖店重点）

**必须验证:**

- [ ] 采集完第一个店铺数据后才能打开店铺列表弹窗获取完整店铺名录
- [ ] 获取店铺列表后必须关闭弹窗，后续每个店铺重新打开弹窗选择
- [ ] 每个店铺切换后必须导航到目标数据页面，不能假设切换后停留在当前页面
- [ ] 店铺列表加载需要滚动加载验证：多次扫描直到无新增店铺
- [ ] 确认触发方式：hover 还是 click，不可假设；搞反了下拉菜单/面板根本不会出现
- [ ] "重定向循环"防护 (P1)：在点击切换店铺按钮后，必须增加强制 URL 变更等待；进入新店铺后，先访问一个非敏感页面确认店铺 ID 已更新，再导航到数据页

**已知坑:**

- 坑1: 店铺列表弹窗是一次性的，关闭后需要重新触发才能打开
- 坑2: 切换店铺后页面可能跳到首页，需强制导航回数据页面
- 坑3: 弹窗中的店铺列表可能因滚动未加载完整，需要多次滚动扫描
- 坑4: 店铺名称类型不止一种，需覆盖专营店、旗舰店、专卖店、企业店等
- 坑5: Session 粘滞导致的重定向循环：如果上一个店铺的 Session 尚未在后端完全销毁，立即切换会导致页面进入 login -> callback -> index -> login 的无限死循环。需在切换后先验证店铺 ID 变更，不可直接访问数据接口

**不可行方案:**

- 通过修改URL参数切换店铺 --> 店铺状态由后端session维护
- 在店铺列表弹窗中连续点击多个店铺 --> 每次只能切换一个
- 假设切换店铺后仍在当前页面 --> 绝大多数情况会被重定向
- 回到首页再切店铺 --> 运营弹窗干扰严重

### 8.3 数据完整性验证

- [ ] 余额和保证金都成功获取才标记"成功"
- [ ] 只拿到一个标记"部分成功"
- [ ] 都没拿到标记"失败"，并记录错误信息
- [ ] 提取到的数据需做合理性检查（如金额不能为负数、不能超过合理范围）

## 九、异常处理约束

### 必须验证

- [ ] 单个账号失败不能中断整个流程，必须继续处理下一个账号
- [ ] 每次失败必须记录足够的调试信息（截图、页面文本、错误信息），用于事后分析
- [ ] 登录失败和数据获取失败需区分处理，不可混为一谈
- [ ] 导航失败（点击菜单未跳转）和数据获取失败需区分处理
- [ ] 弹窗关闭/页面操作时需考虑页面正在导航的竞态条件，关键操作需要异常保护
- [ ] 执行上下文销毁（页面导航中 evaluate 崩溃）必须在关键操作处有 try/except 保护，不可只靠操作前检查；页面可能在任意时刻开始导航，无法通过前置条件完全规避
- [ ] 连续失败熔断：如果连续 3 个店铺出现相同类型的 ElementHandle Check Failed 或导航失败，大概率是平台 UI 大改或 IP 被风控，应立即物理停机（关闭浏览器进程）并通知操作者，避免盲目重试导致封号

### 禁止假设

- [x] 禁止假设所有账号都能成功登录 --> 需设计失败跳过机制
- [x] 禁止假设页面结构永久不变 --> 需保存调试信息，便于页面变化时快速定位问题
- [x] 禁止假设网络始终稳定 --> 需设置合理的超时和重试机制
- [x] 禁止假设evaluate操作一定能在当前页面执行 --> 页面可能在导航中导致上下文被销毁

## 十一、扩展适配检查清单

- [ ] 是否需要点击多个菜单/标签才能看到数据？
- [ ] 页面是SPA还是传统多页？
- [ ] 数据是表格还是卡片形式？

- [ ] 数据默认是否隐藏？需要什么交互才能显示？
- [ ] 同一数据在不同页面是否存在格式差异？（如全角/半角符号、单位差异）

- [ ] 不同账号/店铺的页面布局是否一致？宽度是否有差异？
- [ ] 左侧菜单的点击是传统页面跳转还是SPA局部更新？
- [ ] 点击菜单后页面加载需要等待多久？使用什么等待策略最稳定？
- [ ] 是否存在运营促销弹窗干扰正常操作？

### 弹窗处理环节（新增）

- [ ] 是否存在运营促销弹窗？在哪些页面出现？
- [ ] 是否存在业务功能弹窗（如选择器、下拉菜单）？如何与促销弹窗区分？
- [ ] 弹窗关闭是否会影响正常的hover/click交互状态？

### 店铺/账号切换环节（新增）

- [ ] 是否需要跨店铺切换？切换方式是什么？（hover/click/URL）
- [ ] 切换店铺后页面会停留在原页面还是跳转到其他页面？
- [ ] 店铺名称有哪几种命名规则？是否需要覆盖全部类型？

### 反爬策略

- [ ] 是否有专门的反爬检测？检测哪些维度？
- [ ] 是否需要特殊的浏览器参数？
- [ ] 是否有请求频率限制？
- [ ] 长时间运行是否会触发额外验证？
- [ ] 是否有 Canvas/WebGL 指纹检测？是否需要注入指纹？
- [ ] 是否有鼠标轨迹检测？是否需要贝塞尔曲线模拟？
- [ ] 是否对 CDP 连接敏感？如抖店/字节系平台
- [ ] 多个账号是否需要在独立的浏览器进程中运行以隔离指纹？

### 已知不可行方案（通用）

| 方案                  | 不适用原因               | 适用场景          |
| ------------------- | ------------------- | ------------- |
| 无头模式(headless=True) | 现代网站普遍检测            | 简单静态页面        |
| 纯自动化工具默认配置          | 特征明显易被检测            | 内部系统          |
| 自动验证码识别（通用型）        | 复杂验证码成功率低           | 简单数字/字母验证码    |
| 单一IP高频请求            | 触发IP封禁              | 需配合代理池        |
| 直接URL访问敏感页面         | 可能进入同名但不同的业务模块      | 需从主站导航进入      |
| 统一使用"等待网络空闲"        | 部分页面有持续后台请求导致永远无法空闲 | 需根据页面特征选择等待策略 |
| 基于固定坐标的元素点击         | 不同店铺/会话页面布局可能不同     | 需多策略兜底        |
| 通用弹窗关闭覆盖业务弹窗        | 区分不清会导致功能性弹窗被误关     | 需按弹窗类型分类处理    |
| 通过URL参数切换店铺         | 店铺状态由后端session维护    | 必须通过页面交互切换    |
| display:none强制隐藏弹窗  | 会破坏hover/click的交互状态 | 只能用于最终兜底      |

## 十四、约束更新指南（供 AI 参考）

当 AI 在探索模式中发现某条约束已失效时，应按照以下格式更新文档：

### 更新原则
- P0 硬约束的修改必须在确认"平台底层机制变更"后才能进行，不可仅凭偶然成功就降级
- P1 降级为 P2 最常见（平台改版后旧方案不再必要），P2 可直接删除或标记失效
- 每次更新必须在约束文本后追加验证日期和新发现描述

### 失效标记格式

**P1 降级为 P2 示例：**
```markdown
**旧约束（已失效）**：禁止假设 URL 直接访问财务模块（验证于 2026-05-08）
- 新发现：2026-07-15 重新验证，该平台已改版，URL 直接访问现在可以进入正确模块
- 建议更新为：P2 - 优先使用 URL 直接访问，速度快，需验证页面内容兜底
```

**P2 直接删除示例：**
```markdown
**旧约束（已失效）**：必须通过左侧菜单点击导航（验证于 2026-05-08）
- 新发现：2026-09-01 验证，新版本已移除左侧菜单，改为顶部导航栏。该约束已完全过时。
- 建议：删除此约束
```

**新增约束示例：**
```markdown
**新约束（2026-08-20 新增）**：P1 - 页面增加二次验证弹窗，必须在数据提取前检测并关闭
- 触发条件：连续切换超过 5 个店铺后触发
- 关闭方式：点击"继续操作"按钮
```

### AI 的职责边界
- AI 发现约束失效后，应在本次会话中向用户报告，并建议更新
- AI 不可在没有用户确认的情况下直接修改 P0/P1 约束
- P2 约束的失效可以在用户授权探索模式后直接更新
